[23]: 1. Introducción
[24]: 1.1 Propósito
[25]: Este documento consolida la especificación de requisitos del microservicio ms-notificaciones [NOT], componente transversal del ERP Universitario de la Universidad del Valle - Sede Caicedonia. El servicio centraliza el envío de comunicaciones generadas por otros microservicios y, en la implementación actual, expone una API REST para enviar correos electrónicos mediante SMTP.
[26]: La especificación se construyó a partir del PDF de requisitos funcionales del ERP, el documento consolidado de ms-auditoria usado como ejemplo y el código fuente funcional del proyecto notificacion-service.
[27]: 1.2 Alcance
[28]: Recepción de solicitudes de notificación mediante API REST.
[29]: Envío de correos electrónicos con destinatario, asunto y cuerpo.
[30]: Configuración SMTP para ambientes de desarrollo y ejecución.
[31]: Integración simple desde otros microservicios mediante HTTP/JSON.
[32]: Definición de evolución para plantillas, reintentos, envío masivo, auditoría y salud del servicio.
[33]: 1.3 Fuera del Alcance Actual
[34]: Persistencia de notificaciones en base de datos.
[35]: Gestión completa de plantillas y preferencias de usuario.
[36]: Canales SMS, push o notificación interna.
[37]: Autenticación inter-servicio y trazabilidad distribuida implementadas como middleware.
[38]: Panel administrativo o interfaz gráfica.

[65]: 5. Reglas de Negocio
[66]: Toda notificación por correo debe contener destinatario, asunto y cuerpo.
[67]: El destinatario debe corresponder a una dirección de correo válida.
[68]: El servicio no debe exponer credenciales SMTP en respuestas, logs ni documentación pública.
[69]: Los errores del proveedor de correo deben traducirse a respuestas controladas.
[70]: Plantillas, preferencias, prioridades y reintentos quedan definidos como evolución alineada con el ERP.
[71]: 6. Restricciones del Sistema
[72]: Implementación actual en Java 17 y Spring Boot 2.6.7.
[73]: Endpoint activo en el puerto configurado server.port=8017.
[74]: El envío depende de conectividad hacia el servidor SMTP configurado.
[75]: La versión actual no persiste historial de notificaciones ni reintentos.
[76]: La configuración sensible debe migrarse a variables de entorno o gestor de secretos antes de compartir el repositorio.
[77]: 7. Documentación del Sistema
[78]: El repositorio incluye un README con requisitos previos, configuración, ejecución local, contrato del endpoint y ejemplo de consumo desde Python. Esta documentación debe mantenerse sincronizada con cualquier cambio de API o configuración.

[88]: 8.3 Flujo de un Request
[89]: Un microservicio consumidor envía POST /notifications con cuerpo JSON.
[90]: Spring MVC deserializa el JSON en NotificationRequest.
[91]: NotificationController invoca NotificationService.sendNotification.
[92]: NotificationService crea SimpleMailMessage y llama JavaMailSender.send.
[93]: El proveedor SMTP acepta o rechaza el envío.
[94]: El controlador responde 200 si fue exitoso o 500 si capturó una excepción.
[95]: 8.4 Endpoints Implementados y Propuestos
[96]: 9. Vista de Caso de Uso
[97]: 10. Vista Lógica
[98]: NotificationController depende de NotificationService mediante inyección de dependencias. NotificationService depende de JavaMailSender, bean provisto por Spring Boot a partir de las propiedades spring.mail.*. NotificationRequest se usa como DTO de entrada.
[99]: 11. Vista de Proceso
[100]: El proceso principal es síncrono: la petición HTTP espera a que el servicio SMTP acepte o rechace la operación. Para envío masivo, reintentos y prioridades, se recomienda evolucionar hacia procesamiento asíncrono con cola o scheduler interno.
[101]: 12. Vista de Datos
[102]: La implementación actual no define entidades JPA ni tablas propias para notificaciones. El dato operativo viaja como DTO en memoria y se entrega al servidor SMTP. La propuesta original contempla persistir notificaciones, plantillas, preferencias e historial de reintentos como evolución.
[103]: 13. Seguridad
[104]: Las credenciales SMTP deben cargarse desde variables de entorno o un gestor de secretos.
[105]: No se deben imprimir destinatarios sensibles, credenciales ni cuerpos completos en logs de producción.
[106]: El endpoint debe incorporar autenticación inter-servicio antes de exponerse fuera de un entorno local.
[107]: Las respuestas de error deben evitar filtrar detalles internos del proveedor SMTP.
[108]: Se recomienda rotar las credenciales si estuvieron expuestas en archivos de configuración.
[109]: 14. Diseño de Integración
[110]: El contrato entrante es HTTP/JSON. Los consumidores deben construir un payload con to, subject y body y enviarlo a la URL base configurada del servicio NOT. La respuesta HTTP 200 indica aceptación del envío; los códigos de error deben registrarse en el flujo consumidor.
[111]: 15. Especificación de API REST
[112]: 16. Testing y DevOps
[113]: La versión actual incluye pruebas unitarias y de controlador. NotificationServiceTest verifica que JavaMailSender.send sea invocado una vez; NotificationControllerTest valida que POST /notifications retorne HTTP 200 con un payload básico. Se recomiendan pruebas de validación de campos, escenarios de error SMTP e integración con MailHog.
[114]: Ejecutar localmente con .\mvnw.cmd spring-boot:run en Windows o ./mvnw spring-boot:run en Linux/Mac.
[115]: Empaquetar con Maven y ejecutar el JAR generado cuando se requiera despliegue manual.
[116]: Configurar server.port=8017 o variable equivalente según el ambiente.
[117]: Separar perfiles de desarrollo, pruebas y producción.
[118]: Usar MailHog para pruebas locales cuando no se quiera enviar correo real.
[119]: 17. Justificaciones Técnicas
[120]: Spring Boot permite exponer una API REST mínima con bajo esfuerzo de configuración.
[121]: JavaMailSender abstrae la comunicación SMTP y facilita pruebas con mocks.
[122]: Separar controlador, servicio y DTO mantiene responsabilidades claras.
[123]: Maven Wrapper mejora reproducibilidad en equipos distintos.
[124]: La evolución hacia plantillas, persistencia y reintentos debe conservar el endpoint simple actual para compatibilidad.
[125]: Apéndice A: Guía de Integración
[126]: Para solicitar una notificación, el consumidor debe enviar POST a http://localhost:8017/notifications con Content-Type application/json. El cuerpo debe incluir to, subject y body. El consumidor debe manejar errores HTTP 4xx/5xx y registrar el resultado dentro de su propio flujo de negocio.
[127]: Nota de seguridad: este documento no incluye valores reales de usuario o contraseña SMTP. En ambientes compartidos, las credenciales deben reemplazarse por variables de entorno y rotarse si estuvieron expuestas en archivos de configuración.
